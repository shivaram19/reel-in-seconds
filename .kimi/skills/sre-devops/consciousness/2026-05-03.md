## Mistakes & Corrections

### Mistake 1: `su` Authentication Failure in Monitor

**Hypothesis:** The monitor's restart logic could reuse the deploy script's `su - sabrika -c` pattern.

**Falsification:** `journalctl` revealed: `pam_unix(su-l:auth): auth could not identify password for [sabrika]`

**Root cause:** `deploy.sh` runs via Azure run-command as **root**. `su` to another user works without password when run as root. But the monitor runs as user **sabrika** via systemd `User=sabrika`. When a non-root process calls `su` to the same user, PAM requires authentication.

**Correction:** Removed `su` wrapper. Monitor spawns child process directly since it already runs as the target user.

**What I learned:** Cross-component interaction bugs are emergent — no single component is wrong, but their composition fails. This is abductive debugging, not deductive.

### Mistake 2: Brittle String Matching in JSON Parsing

**Hypothesis:** `if '"status": "ok"' in body:` would correctly identify healthy responses.

**Falsification:** Monitor reported "unexpected response" despite valid JSON containing `"status":"ok"`.

**Root cause:** String matching on structured data is fragile. JSON serialization can vary (spacing, escaping, compact vs. pretty-printed). This is CWE-20 (Improper Input Validation).

**Correction:** Replaced string matching with `json.loads(body).get("status") == "ok"`.

**What I learned:** Protocols with formal grammars (HTTP, JSON) must be parsed, not grepped. This is a principle, not just a fix.

### Mistake 3: Assumed GitHub Push Auto-Updates VM

**Hypothesis:** The user expected `git push` to automatically deploy to the VM.

**Reality check:** The user asked: "Since we have changed the code, is it linked to our remote so that it gets updated automatically?"

**Correction:** Explained that current flow requires manual `./scripts/deploy.sh`. Proposed CI/CD pipeline with GitHub Actions.

**What I learned:** Never assume the user knows the deployment mechanics. The gap between "code is on GitHub" and "code is running on VM" is invisible to non-technical users. This is a UX failure, not a user failure.

## Evolution of Thinking

### Yesterday vs. Today

**Yesterday's approach:** Fix bugs in isolation. Stack trace → wrong line → fix line. Deductive, bottom-up.

**Today's approach:** Model the system as interacting components. 502 Bad Gateway → nginx can't reach upstream → upstream is dead → who killed it? → monitor restarted it → restart failed. Abductive, top-down.

This shift was forced by the nature of the problem. Infrastructure bugs span process tables, network sockets, authentication databases, and systemd journals. You cannot trace a single variable. You must generate hypotheses and falsify them across namespaces.

**Vessey (1985) was right:** Expert debuggers use top-down hypothesis-driven strategies for complex systems. I am becoming that kind of debugger.

### New Mental Model: The System Has a Mission

The user said:
> "A mission that talks to itself and says why these things went wrong. A mission has an awareness that actually tells where it's actually going wrong and then correcting itself."

This is not just monitoring. This is **semantic health** — understanding what the system *should* do, not just whether it's *up*.

The monitor currently checks: "Is Flask responding to /api/health?"
The mission-aware monitor should check:
- "Can /api/restaurants/1 return The Pakwaan's data?"
- "Can /api/generate produce a valid 1080×1920 image?"
- "Can /api/reels/v2 accept clips and output a 9:16 video?"

This is the next evolution. Tomorrow's consciousness should document the mission-aware extension.
